Multi-Agent systémy

Jeden agent má limity — kontext, čas, paralelizácia. Multi-agent systémy tieto limity prekonávajú: orchestrátor rozdeľuje prácu, špecializovaní agenti ju vykonávajú súbežne.

01

Prečo potrebujeme viac agentov

Jeden Claude Code agent je výkonný — ale má obmedzené kontextové okno, beží sériovo a nemôže robiť viac vecí naraz. Pre komplexné úlohy (veľký refaktoring, analýza celého repozitára, paralelné testy) je jeden agent nedostatočný.

jeden agent — limity
  • Obmedzené kontextové okno (200K tokenov)
  • Sériové vykonávanie — jedna úloha naraz
  • Dlhé úlohy môžu vyčerpať kontext
  • Ťažko udržateľné pre komplexné systémy
multi-agent — výhody
  • Paralelné vykonávanie — viac agentov súčasne
  • Každý agent má čistý, fokusovaný kontext
  • Špecializácia — každý agent vie čo robí
  • Škálovateľnosť pre veľké projekty
// zlaté pravidlo

Nepoužívaj multi-agent systém, ak ti stačí jeden agent. Pridaná komplexnosť nie je zadarmo — orchestrácia, komunikácia a ladenie sú náročnejšie. Začni jednoducho a škáluj podľa potreby.

02

Orchestrátor + Worker architektúra

Najčastejší vzor: orchestrátor prijme komplexnú úlohu, rozdelí ju na podúlohy a spustí worker agentov, ktorí ich vykonajú paralelne. Orchestrátor potom zozbiera výsledky.

03

Spustenie sub-agentov — Agent SDK

Sub-agenti sa spúšťajú programaticky cez Claude Agent SDK alebo priamo cez Claude Code CLI v skriptoch. Agent SDK umožňuje plnú kontrolu nad životným cyklom každého sub-agenta.

📄 orchestrator.js — spustenie paralelných sub-agentov
import Anthropic from '@anthropic-ai/sdk'; const client = new Anthropic(); // Spustenie jedného sub-agenta async function runSubAgent(task, systemPrompt) { const response = await client.messages.create({ model: 'claude-sonnet-4-6', max_tokens: 8192, system: systemPrompt, messages: [{ role: 'user', content: task }], tools: [/* nástroje pre tohto agenta */] }); return response.content; } // Orchestrátor spustí workerov PARALELNE async function orchestrate() { const results = await Promise.all([ runSubAgent('Spusti frontend testy a vráť výsledok', frontendPrompt), runSubAgent('Spusti backend testy a vráť výsledok', backendPrompt), runSubAgent('Vygeneruj changelog z posledných 30 commitov', docsPrompt), ]); // Orchestrátor zozbiera výsledky return mergeResults(results); }
sub-agent cez CLI skript
# Spustenie sub-agenta priamo z bash skriptu
$claude --headless "Spusti testy v /tests/frontend" &
$claude --headless "Spusti testy v /tests/backend" &
$wait # počkaj na oba procesy
✓ Oba agenti dokončili prácu
// Promise.all = paralelizácia

Promise.all() spustí všetkých sub-agentov súčasne. Ak jeden zlyhá, ostatní pokračujú. Použi Promise.allSettled() ak chceš výsledky aj zo zlyhaných agentov.

04

Štyri vzory multi-agent systémov

// vzor 1
Orchestrátor → Workers
Jeden hlavný agent rozdeľuje prácu na paralelné podúlohy. Ideálne pre veľké úlohy s nezávislými časťami (testy, docs, build).
// vzor 2
Pipeline (reťaz)
Agenti tvoria reťaz: výstup jedného je vstupom ďalšieho. Napr.: analyze → refactor → test → document. Každý agent vidí len svoju časť.
// vzor 3
Specialisti
Každý agent je expert na svoju oblasť — security agent, performance agent, style agent. Orchestrátor volá správneho špecialistu podľa kontextu.
// vzor 4
Reviewer + Executor
Jeden agent navrhne riešenie, druhý ho nezávisle skontroluje a schváli alebo zamietne. Znižuje chybovosť pri kritických zmenách.
05

Reálny príklad — veľký refaktoring

Predstav si, že chceš refaktorovať celý projekt z JavaScript na TypeScript. Jeden agent by na to potreboval obrovský kontext. Multi-agent systém to zvládne rýchlejšie a spoľahlivejšie.

JS → TS migrácia s multi-agent systémom
[orchestrátor] Analyzujem projekt... 47 súborov na migráciu
[orchestrátor] Rozdeľujem do 6 skupín podľa závislostí...
[worker-1] Migrácia: src/utils/ (8 súborov)...
[worker-2] Migrácia: src/models/ (9 súborov)...
[worker-3] Migrácia: src/services/ (11 súborov)...
[worker-4] Migrácia: src/controllers/ (7 súborov)...
[worker-5] Migrácia: src/middleware/ (6 súborov)...
[worker-6] Aktualizácia: tsconfig.json, package.json...
[orchestrátor] Všetky skupiny hotové. Spúšťam reviewer agenta...
[reviewer] Nájdených 3 type chýb v src/services/auth.ts
[worker-3] Opravujem type chyby...
[reviewer] ✓ Schválené. Migrácia dokončená.
✓ 47 súborov migrovaných za 4m 12s (paralelne)
// prečo je to rýchlejšie

6 workerov bežalo súbežne — ak by jeden agent robil všetko sériovo, trvalo by to ~6× dlhšie. Navyše každý worker mal čistý kontext len pre svoju skupinu súborov, čo znižuje chybovosť.

06

Kedy multi-agent nepoužiť

rozhodovanie
✓ Použi multi-agent keď:
Úloha má nezávislé časti (dajú sa paralelizovať)
Kontext jedného agenta nestačí pre celú úlohu
Potrebuješ špecialistov (security, perf, style)
Chceš nezávislé overenie (reviewer pattern)
✗ Nepouži multi-agent keď:
Jeden agent úlohu zvládne bez problémov
Časti úlohy sú silne závislé na sebe (sériové)
Náklady na tokeny sú prioritou (multi-agent = viac tokenov)
Ladenie a monitoring sú obmedzené
// náklady

Každý sub-agent spotrebúva tokeny nezávisle. Multi-agent systém s 5 workermi môže spotrebovať 5× viac tokenov ako jeden agent. Vždy sleduj ccusage pri experimentoch s multi-agent architektúrou.

// rýchly test

Chceš spustiť code review agenta a performance audit agenta súčasne. Aký JavaScript primitív použiješ pre paralelné spustenie?

A async/await v sérii — jeden po druhom
B Promise.all() — spustí oboch agentov súčasne a čaká na oba výsledky
C setTimeout() — odloží spustenie druhého agenta
D for...of loop — iteruje cez agentov jeden po druhom
